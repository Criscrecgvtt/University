Servidor Asincrónico

  • Dirigido por eventos. (nodeJS, Async .NET)
  • Programa = conjunto de acciones para responder a diferentes tipos de eventos
  • La llegada de un evento activa las acciones asociadas
    • Si no estamos ejecutando otra cosa, ejecutamos la acción
    • Si estamos ejecutando otra cosa, guardamos la acción en una cola
    • Cola de eventos = acciones activadas pero pendientes de ejecución
  • múltiples actividades, pero no acceden de forma concurrente al estado compartido
    • Ventaja.- No hay estado compartido: $\downarrow$ complejidad $\uparrow$ escala mejor
    • Ventaja.- Modelo más próximo a la forma real de trabajo $\rightarrow$ dirigido por eventos
    • facilita el razonamiento sobre la corrección del código
    • Inconveniente.- El orden de activación (orden en cola de eventos) puede ser importante
    • Necesita una gestión adecuada del estado al implantar las acciones
    • Inconveniente.- Todo el entorno debe ser asincrónico

Ejemplo: servidor Echo (Servidor Asincrónico en NodeJS)

// fichero echoServer.js
const net = require('net')
let server = net.createServer(onClientConnected) // Create Server instance 
server.listen(8080, 'localhost', () => {console.log(`server listening on ${server.address()}`)})

function onClientConnected(client) {  
    let remoteAddress = `${client.remoteAddress}:${client.remotePort}`
    console.log(`new client connected: ${remoteAddress}`)

    client.on('data', (data) => {
        console.log(`${remoteAddress} Recv: ${data}`)
        client.write(`Echo: ${data}`)
    })
    client.on('close', ()    => {console.log(`connection from ${remoteAddress} closed`)})
    client.on('error', (err) => {console.log(`Connection ${remoteAddress} error: ${err.message}`)})
}

// fichero echoClient.js
var net = require('net')
var nick = process.argv[2] || 'client' // accede a argumento en linea de órdenes:  en process.argv tenemos ['node', 'echoClient', 'idCliente'] 
var client = net.connect({port: 8080}, function() {console.log(nick+' connected to server!')})

process.stdin.resume(); process.stdin.setEncoding('utf8')  // inicializa entrada estardrd (teclado)

    // acciones asociadas a eventos de teclado
process.stdin.on('data', (str)=> {client.write(nick+': '+str)})
process.stdin.on( 'end', ()   => {client.write('connection closed')})

    // acciones asociadas al canal de comunicación con el servidor
client.on('data', (str)=> {console.log(str.toString())})
client.on( 'end', ()   => {console.log('disconnected from server')})

Servidores asincrónicos.- Cuestiones

  1. El ejemplo de servidor asincrónico no tiene bucles. ¿Cuando acaba el programa?
  2. El servidor asincrónico dispone solo de un hilo. ¿Qué pasa si mientras ejecuta código llega un evento?

Servidor asincrónico.- Ejercicio

Lanza en terminales distintos las siguientes órdenes

  • node echoServer
  • node echoClient A
  • node echoClient B

Responde a las siguientes cuestiones:

  • Indica si el servidor puede atender simultáneamente a más de un cliente (atender a un segundo cliente cuando todavía no ha terminado el primero). ¿Cómo se asocia cada respuesta al cliente correcto?
  • Cada vez que escribas un texto en el terminal del cliente dicho texto debe llegar al servidor y contestar a ese cliente. Compruébalo
  • ¿Qué pasa cuando el primer cliente termina con control-C? ¿Y si también finaliza el segundo?. Indica cómo afecta todo eso al servidor
  • control-C permite abortar la ejecución, pero control-D (en Linux) permite cerrar la salida estandard. ¿Qué pasa con los clientes y el servidor si en los clientes se usa control-D en lugar de control-C?
